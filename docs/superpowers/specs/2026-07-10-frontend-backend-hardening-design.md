# 前后端体验优化与 Go 后端加固设计

日期：2026-07-10
状态：待用户书面审核
目标版本：Go 1.26.4、React 19、Vite 7、Astryx Design

## 1. 背景与目标

当前项目由同一套 React 前端承载三种产品界面：服务端公共商店、服务端管理后台和独立客户端；后端则由服务端与客户端两个 Go 二进制分别提供 API、调度、存储和嵌入式前端。

本轮工作的目标不是重新发明产品或设计系统，而是在保留现有功能和未提交开发成果的前提下完成两类改进：

1. 让三种前端界面的主任务、信息层级、反馈和移动端交互更清楚、更迅速。
2. 修复 Go 后端审计中发现的生命周期、逻辑并发、资源上限、内存峰值和工程验证缺口。

最终结果应满足：用户能更快完成浏览、安装和管理任务；服务端能在并发、长连接、大文件和关闭场景下保持可预测；项目在 CI 中持续验证 race、lint、依赖整洁度和构建产物。

## 2. 已确认的方向

采用“全覆盖、分阶段、并行实施”方案：

1. 建立共享交互基础。
2. 优化服务端公共商店。
3. 优化独立客户端。
4. 优化服务端管理后台。
5. 同期推进后端运行时与并发加固。
6. 后端正确性稳定后推进迁移内存和列表性能优化。
7. 最后统一完成现代 Go 清理、CI 门禁、生成产物和跨端验收。

前后端可并行开发，但同一高冲突文件只有一个集成所有者。`client/src/App.tsx`、全局样式、语言包、Go 路由装配和生成产物不允许多个并行任务同时编辑。

## 3. 范围

### 3.1 前端范围

- 服务端公共商店：首页、发现、分类、应用卡片、详情、软件下载与软件源订阅。
- 独立客户端：应用发现、软件源、安装状态、已安装、历史、设置和详情页。
- 服务端后台：审核、站点、用户与群组、分类标签、集合、存储、备份和迁移。
- 共享体验：导航、按钮反馈、加载、空状态、错误恢复、Toast、表单保存状态、移动端和减少动态效果。

### 3.2 后端范围

- 两个 Go 二进制的启动、信号处理、HTTP 超时和有序关闭。
- SSE 长连接写入与断开处理。
- 软件源密码轮换的原子性。
- `singleflight` 请求取消和后台加载生命周期。
- 客户端自动同步 goroutine 的跟踪和关闭。
- 数据库连接池配置。
- JSON、软件源响应、迁移包和外部 HTTP 的资源限制。
- 迁移与备份的流式化。
- 应用列表统计的基准、查询次数和分配优化。
- Go 1.26 现代写法、lint、race、vet、tidy 和覆盖率门禁。

### 3.3 不在本轮范围

- 替换 React、Vite、Astryx、Ent 或数据库技术栈。
- 改变软件源 v1/v2 的既有公开语义。
- 引入分布式任务系统、Redis 或外部消息队列。
- 为尚无实测瓶颈的代码引入 `sync.Pool`、分片锁或无锁结构。
- 大规模重命名全部 API、数据库字段或路由。
- 对生成的 Ent 文件做手工修改。

## 4. 前端设计

### 4.1 共享交互基础

所有界面沿用 Astryx 组件和现有主题变量，不新增第二套设计系统。

共享规则：

- 每屏只有一个视觉上的主操作。
- 所有可按压控件提供 100–160ms 的按压反馈，缩放范围为 0.97–0.99。
- 高频导航、键盘操作和标签切换不做位移动画；必要时仅做不超过 160ms 的透明度反馈。
- 弹窗和偶发面板使用 180–240ms 的强 `ease-out`，从 `scale(0.96–0.98)` 与透明度进入，不从 `scale(0)` 出现。
- 只动画 `transform`、`opacity`、颜色和边框颜色。
- 悬浮效果只在 `hover: hover` 且 `pointer: fine` 时启用。
- `prefers-reduced-motion` 保留颜色和透明度反馈，移除位移与缩放运动。
- Toast 用于短暂确认；表单错误、迁移结果和安装失败必须保留就地状态及恢复动作。
- 所有保存、安装、同步和删除操作有 pending 状态，并防止重复提交。
- 空状态只提供一个最可能的下一步动作。

### 4.2 服务端公共商店

公共商店的首要任务是发现应用，其次是了解并订阅软件源。

首页调整：

- Hero 只保留“发现应用”为主操作。
- 登录或提交应用作为次级入口，不与发现争夺视觉优先级。
- 软件源地址从统计卡片中移出，独立为“订阅此商店”面板，提供复制、版本和简短使用说明。
- 纯统计卡片不提供悬浮位移，避免让不可点击内容看起来可操作。
- 广告继续作为可选内容，不得挤压主操作或导致移动端首屏只剩广告。

应用浏览调整：

- 卡片只显示名称、摘要、分类、版本、评分或下载量中的必要信息，以及一个主操作。
- 校验和、审核、安装保护等信任信息集中到详情页，不在卡片上堆叠全部徽章。
- 分类、排序和搜索位于结果之前；当前条件和结果数量始终可见。
- 卡片整体打开详情，下载按钮阻止冒泡并表达独立命令。
- 应用详情承担截图、版本、校验和、下载来源、评论和管理信息。

### 4.3 独立客户端

客户端的默认任务是发现并安装应用，软件源管理是必要但较安静的配置任务。

- 已存在可用软件源时默认进入发现页。
- 没有软件源时显示首次引导，并把“添加软件源”作为唯一主操作。
- 软件源卡片采用紧凑状态行，清楚显示同步状态、最近时间、应用数、群组和错误恢复动作。
- 安装操作根据本地状态显示安装、更新、重新安装或回滚。
- 安装过程展示排队、下载、校验、安装和结果状态，不只依赖 Toast。
- 已安装应用按更新可用性、来源和本地未知来源区分。
- 应用详情保持全页或全屏 sheet，不退回窄侧栏。
- 设置页的高频标签切换即时完成；保存按钮显示未保存、保存中、已保存和失败状态。

### 4.4 服务端管理后台

后台按任务组织，而不是把设置堆在一个长页面中。

- 一级任务保持审核、站点、用户与群组、分类标签、集合、存储、备份和迁移。
- 每个任务页有明确标题、状态摘要和一个主操作。
- 保存型页面显示未保存状态；长表单在桌面端提供稳定可见的保存区，在移动端使用底部安全区域。
- 危险操作使用明确对象名和后果说明，不能只写“确认删除”。
- 列表操作在触屏设备上始终可达，不能只依赖 hover。
- 迁移、备份和存储测试结果使用持久结果面板，显示时间、目标、部分成功和错误恢复路径。
- 管理标签切换不使用每次进入都重复播放的位移动画。

## 5. 后端设计

### 5.1 启动与关闭

两个二进制各保留一个初始化路径，`main.go` 仅负责调用可测试的执行函数并设置退出码。

运行顺序：

1. 从环境加载并验证配置。
2. 打开并配置数据库连接池。
3. 完成 schema 和应用迁移。
4. 创建依赖和后台任务。
5. 启动 HTTP 服务。
6. 监听 `SIGINT`、`SIGTERM` 或迁移替换完成事件。
7. 停止接受新请求。
8. 取消调度器、备份和后台加载。
9. 等待受管 goroutine 退出。
10. 在统一超时内关闭 HTTP、存储和数据库资源。

服务端与客户端都配置 `ReadHeaderTimeout`、`IdleTimeout` 和合理的 header 上限。普通 API 使用有界请求和写入时间；SSE 不继承会在固定时刻终止连接的全局写超时，而是按心跳使用可刷新写期限并在首次写失败时退出。

### 5.2 软件源密码轮换

密码和轮换时间戳必须作为一个逻辑状态原子更新。

- 同一进程内用专用互斥锁合并轮换检查。
- 数据库更新在事务内完成，并检查每次写入错误。
- 多实例部署不是本轮目标，但代码不得向调用方返回未成功持久化的 token。
- 并发测试同时发起多个到期请求，所有成功请求必须观察到同一个最终 token。

### 5.3 后台任务和 singleflight

- 所有显式 goroutine 由 `WaitGroup.Go` 或 `errgroup` 跟踪。
- 调度器关闭必须执行 cancel、停止接收新任务、等待运行中任务、最后关闭数据库。
- `singleflight` 使用 `DoChan`，请求调用方可以在自身 context 取消时退出等待。
- 实际共享加载绑定服务器生命周期，并配置最大执行时间；不得使用无限期的 `context.WithoutCancel`。
- chat fanout 保持非阻塞发送，明确记录“慢订阅者丢事件”的策略；客户端收到任何事件后重新读取权威会话状态，因此允许丢失提示事件而不丢失消息事实。

### 5.4 数据库连接池

通过标准 SQL driver 包装 Ent driver，统一设置连接池。

- SQLite 默认限制写连接并保留少量空闲连接，使用 WAL 与 busy timeout；实际连接数由并发 benchmark 决定。
- PostgreSQL/MySQL 使用可配置的最大连接、空闲连接、最大寿命和空闲寿命。
- 配置必须有安全默认值，并允许环境变量覆盖。
- 启动测试验证配置生效，压力测试验证 SQLite 不出现持续 `database is locked`。

### 5.5 输入、外部 HTTP 和安全边界

- 服务端和客户端 JSON 解码统一限制请求体大小、拒绝未知字段，并拒绝第一个 JSON 值之后的额外内容。
- 软件源 feed 设置最大响应体；图片、评论和聊天代理继续使用更小的独立上限。
- 非流式外部 HTTP 使用共享客户端，配置拨号、TLS、响应头和总超时。
- SSE 代理使用独立的流式客户端，只设置建立连接阶段的超时，不设置总请求超时。
- 软件源 URL 允许 HTTP/HTTPS。考虑到私有软件源可能位于局域网，本轮不直接禁止私网地址；改为明确记录信任模型，并为多用户/OIDC 部署预留“仅管理员可添加私网源”的策略开关。
- 自托管环境缺少会话密钥时不得静默使用公开固定值。LazyCat manifest 的 `stable_secret` 路径保持不变。
- 管理员登录失败状态增加 TTL、容量上限和服务端节流；客户端 CAPTCHA 不能作为唯一安全边界。

### 5.6 迁移和备份内存

迁移与备份不再以完整 `[]byte` 或 `bytes.Buffer` 持有大包。

- 上传通过 `MaxBytesReader` 后流入受控临时文件。
- 临时文件目录、权限、关闭和删除由同一生命周期函数负责。
- preview 和 import 接收 `io.ReaderAt`/文件大小或可重开 reader，不复制完整包。
- export 先写临时文件，成功后再发送响应，避免响应头已发送后无法报告导出失败。
- backup 对同一临时文件计算流式 SHA256，并为每个目标重新打开 reader。
- 同时限制压缩包大小、单条目大小、总解压大小和文件数量，防止 zip bomb。

### 5.7 列表性能

不凭静态猜测直接改查询。先扩充 benchmark：

- 50、100 个应用。
- 每个应用包含标签、版本、评分和大量下载历史。
- 记录 SQL 查询次数、`ns/op`、`B/op` 和 `allocs/op`。
- 使用 `benchstat` 比较变更前后至少 10 次结果。

若数据确认下载周期统计成为热点，则把日、周、月、年四次聚合合并为一次条件聚合或维护可验证的汇总表。若本地化 JSON 解码成为主要分配来源，再考虑在 DTO 构建周期内复用解码结果，而不是引入全局缓存。

## 6. 现代 Go 与工程门禁

目标版本为 Go 1.26.4，允许使用不高于该版本的标准能力。

机械现代化范围：

- benchmark 使用 `b.Loop()` 和 `b.Context()`。
- 测试使用 `t.Context()`。
- `errors.As` 使用 `errors.AsType`。
- 迭代拆分使用 `strings.SplitSeq`/`FieldsSeq`。
- WaitGroup 启动任务使用 `WaitGroup.Go`。
- map key 收集使用 `maps.Keys` 与 `slices.Collect`。
- 在不改变 JSON 空数组/空对象契约的前提下采用 `omitzero`。
- 清除无效赋值、未使用符号和已弃用标准库/AWS SDK API。
- 重要输出文件的 `Close` 错误必须进入返回值；HTTP body 的忽略关闭错误使用显式 `_ =` 表达意图。

CI 增加以下门禁：

1. `go test ./...`
2. `go vet ./...`
3. `go test -race ./...`
4. `golangci-lint run --timeout=5m`
5. `go mod tidy -diff`
6. 前端 `npm ci`、audit 和 build
7. OpenAPI 与 LazyCat YAML 校验

`govulncheck` 可加入独立安全 job；若网络或漏洞数据库不可用，应明确报告为基础设施失败，不能伪装成通过。

## 7. 并行任务边界

### 7.1 并行执行单元

| 任务 | 独占范围 | 不得修改 |
| --- | --- | --- |
| FE-1 服务端前台 | `modules/storefront/**`、`styles/storefront.css` | `App.tsx`、语言包、后台/客户端模块 |
| FE-2 独立客户端 | `modules/client/**`、`styles/client.css` | `App.tsx`、语言包、商店/后台模块 |
| FE-3 管理后台 | `modules/admin/**`、`styles/admin.css` | `App.tsx`、语言包、商店/客户端模块 |
| BE-1 运行时与并发 | `cmd/**`、生命周期、scheduler、singleflight、密码轮换、SSE | 前端文件、迁移格式、统计查询 |
| BE-2 内存与性能 | 迁移、备份、DB pool、app metrics benchmark | 前端文件、认证语义 |
| INT 集成 | `App.tsx`、语言包、共享样式、CI、生成产物 | 未经任务所有者确认的功能内部重写 |

BE-1 与三个前端任务可同时进行。BE-2 在 BE-1 稳定后开始，避免同时修改服务器生命周期与备份关闭路径。现代化和 lint 清理由 INT 阶段统一处理，避免机械修改与功能任务冲突。

### 7.2 共享文件规则

- 并行任务不得直接编辑 `client/src/locales/zh.ts` 与 `en.ts`；提交所需 key 清单，由 INT 一次性合并。
- 并行任务不得构建或提交 `clientembed/dist`、`client/dist`、`web/dist`；只在集成完成后生成一次。
- Ent schema 变化由后端任务所有者完成，生成文件只生成一次并与 schema 同批验证。
- 开始任务前读取当前 diff，保留用户已有未提交改动。
- 任一任务发现必须跨边界修改时，先记录依赖，由集成阶段处理，不抢改共享文件。

## 8. 错误处理与可观测性

- 用户错误返回稳定错误码和可恢复说明，不暴露数据库、路径或远端凭据。
- 后台任务失败记录任务名、触发来源和可操作上下文，不记录密码、token 或完整授权 header。
- 被丢弃的 chat 提示事件使用 debug 级别计数或日志，不在高频路径打印普通 info。
- 迁移和备份返回结构化阶段结果：准备、导出、上传、清理和重启。
- 关闭阶段汇总多个独立清理错误，使用 `errors.Join` 返回，不因第一个错误跳过其余清理。

## 9. 测试与验证

### 9.1 后端新增测试

- 并发密码轮换只生成并持久化一个 token。
- 取消等待中的 singleflight 请求会及时返回，后台加载受服务器 context 控制。
- 调度器 Close 等待启动同步退出，无 goroutine 泄漏。
- SSE 在超过普通 API 写超时时仍可发送心跳，并在写失败后退出。
- JSON 过大、尾随 JSON 和软件源响应过大被拒绝。
- 迁移上传、预览、导入、导出和备份使用临时文件且完成后清理。
- SQLite 并发读写不会持续锁死；数据库连接池设置符合配置。
- 大量下载记录下的列表 benchmark 和查询次数被固定。

并发时间相关测试优先使用 `testing/synctest`；至少一个测试使用 goroutine leak 检测。

### 9.2 前端验证

- 桌面宽度：1440、1024。
- 移动宽度：768、375、320。
- 浅色、深色及至少一个非默认 Astryx 主题。
- 键盘导航、焦点可见性、Escape 关闭、触屏操作可达性。
- `prefers-reduced-motion` 下不出现位移动画。
- 加载、空数据、网络错误、权限不足、部分成功和重复提交。
- 安装、同步、保存、备份和迁移的真实状态转换。

### 9.3 集成门禁

完成条件：

- Go 普通测试、vet、race、lint、tidy 全部通过。
- 后端新增并发测试在 `-race` 下通过。
- 性能变更有变更前后 benchmark 与 benchstat，不接受只凭单次数字的“优化”。
- React TypeScript 构建通过。
- OpenAPI 和 LazyCat 配置校验通过。
- 生成的三个前端 dist 与源码一致。
- 服务端公共商店、独立客户端和管理后台完成桌面与移动端浏览器 smoke test。
- 工作区原有改动均被保留并能解释来源，不混入无关重构。

## 10. 风险与回退

- SSE 超时调整可能改变普通 API 的防慢请求保护，因此必须以“普通 API 有界、SSE 单独策略”实现，不能简单移除所有写超时。
- SQLite 连接池数值必须通过测试确定，不能照搬 PostgreSQL 默认值。
- `omitzero` 可能改变空 slice/map 的 JSON 表现，只对已确认契约安全的字段使用。
- 迁移流式化涉及临时文件清理和多目标重新读取，失败路径必须先有测试。
- 前端简化卡片不能删除用户仍需要的信任信息，只能把它移动到详情页。
- 并行开发的最大风险是共享文件冲突，因此 App 装配、语言包和生成产物始终由 INT 单线程合并。

## 11. 验收标准

- 公共商店首屏的唯一主任务是发现应用，软件源订阅有独立入口。
- 客户端在已有软件源时默认进入发现页，无源时提供明确首次引导。
- 后台每个任务页只有一个主操作，并能持续显示保存或运行结果。
- 高频标签和导航切换没有拖慢体验的位移动画。
- 服务端和客户端收到终止信号后有序退出，不留下受管 goroutine。
- 软件源密码轮换在并发下保持一致。
- SSE 不因普通 API 的固定写超时周期性断开。
- 迁移和备份不再把最高 512 MiB 包复制到多个内存缓冲区。
- 公共 JSON 和软件源响应均有明确大小上限。
- Go 1.26 现代化完成且 `golangci-lint` 零问题。
- CI 持续执行 race、vet、lint 和 tidy 检查。
- 所有性能优化都有可重复 benchmark 证据。
